# 生产环境里的功能流程测试、异步子流程测试、AI 应用测试实践

## 一、先说核心结论

在真实生产环境里，**一个功能流程里包含异步子流程**是很常见的。此时如果测试仍然使用“主流程一返回就立刻断言所有副作用都完成”的方式，通常会出现两类问题：

1. **误报失败**：主流程其实成功了，但异步任务还没来得及落库、写事件、写记忆、发消息。
2. **结果抖动**：同一条测试有时通过、有时失败，难以定位，稳定性很差。

所以在实际里，通常不会把“同步结果”和“异步副作用”混成一次立即断言，而是会拆层处理。

---

## 二、实际中怎么测带异步子流程的功能流程

### 1. 拆成两层断言

最常见的方法是把测试拆成两层：

#### A. 主流程即时断言
只验证调用返回后**立刻应该成立**的东西，例如：
- 请求是否成功
- 主回答是否生成
- 主记录是否持久化
- 前台可见状态是否正确
- 是否正确投递了异步任务

#### B. 异步结果最终断言
再单独验证**允许延迟完成**的副作用，例如：
- 异步写库是否完成
- 消息队列消费者是否处理完成
- 记忆写入是否完成
- 搜索索引是否更新
- 通知是否发送

这种方式本质上是：

> 把“是否成功触发异步流程”和“异步流程最终是否完成”拆开测。

这比“一步断言所有都完成”稳定得多。

---

### 2. 引入“最终一致性等待”机制

对于异步结果，实际中常用的不是立即断言，而是：

- 轮询查询
- 有超时上限
- 满足条件就提前结束
- 超时才判失败

典型伪代码：

```python
for _ in range(30):
    state = query_async_result()
    if state == "done":
        break
    time.sleep(1)
assert state == "done"
```

这类方式在工程里通常叫：
- `eventually`
- `wait_until`
- `assert_with_retry`
- `poll_until`

它解决的是：

> 异步系统不是“现在就对”，而是“在合理时间内最终对”。

---

### 3. 测“事件是否发出”，而不是一开始就测“最终副作用”

在很多生产测试里，异步链路的第一目标不是“立刻看到副作用”，而是先确认：

- 事件是否被正确发布
- job 是否被正确提交
- 队列消息是否被正确入队
- callback 是否被正确注册

也就是说，主流程测试常常只验证：

> 异步链路被正确触发了。

之后再由：
- worker 集成测试
- consumer 集成测试
- 端到端延迟测试

去验证最终副作用。

这种拆法比所有事都压在一个 case 上更可维护。

---

### 4. 给异步流程增加“可观测点”

如果系统没有可观测点，异步测试就很难稳定。

实际工程里会有意识增加这些观测手段：
- job id
- event id
- task state
- callback state
- 阶段状态字段
- audit log
- trace/span
- 中间落盘标记

例如对 memory 写入来说，最好能观察到：
- 是否提交了异步 memory job
- job 是否开始处理
- job 是否结束
- 是否成功写入 session scope
- 写入失败原因是什么

如果没有这些观测点，测试只能通过外部副作用猜测系统状态，误差会很大。

---

### 5. 把异步测试按稳定性分层

实际生产中，通常不会把所有异步检查都放在同一层。

#### 单元测试
- 测函数逻辑
- 不真的起异步 worker
- 用 mock 验证“是否提交任务”

#### 集成测试
- 起真实依赖或半真实依赖
- 验证任务能跑完
- 验证库、缓存、索引、副作用是否最终一致

#### 端到端测试
- 测整个链路
- 允许更长超时
- 数量少，但覆盖关键路径

这是工业实践里最常见的分层方式。

---

## 三、对于 AI 应用，为什么测试更容易出问题

AI 应用和普通业务系统相比，会多出几类特殊问题：

1. **输出非确定性**
2. **模型版本变化带来结果漂移**
3. **多轮上下文会放大前一轮误差**
4. **异步子流程更多**，如：记忆写入、索引更新、评估任务、消息回调
5. **“对不对”往往不是布尔值**，而是有灰度区间

所以 AI 测试一般不会只靠“精确字符串相等”，而要采用更分层的方法。

---

## 四、AI 应用测试在实际里通常怎么分

### 1. 结构正确性测试
测的是：
- 有没有返回
- 有没有调不该调的工具
- JSON 是否合法
- 是否包含必须字段
- 状态流是否正确

这是最稳定的一层。

### 2. 行为约束测试
测的是：
- 是否命中了正确工具
- 是否避免了不必要工具调用
- 是否使用了记忆
- 是否遵守了安全约束
- 是否遵守了格式要求

这层通常比纯文本匹配更稳。

### 3. 语义结果测试
测的是：
- 回答是否覆盖核心信息
- 是否答非所问
- 是否包含关键概念
- 是否遗漏关键约束

这里通常不会要求全文一致，而会要求：
- 包含关键点
- 不包含明显错误
- 最小质量阈值

### 4. 端到端任务测试
测的是：
- 完整任务是否完成
- 用户目标是否达成
- 系统整体体验是否可接受

这层一般 case 少，但价值高。

---

## 五、多轮对话测试在实际中怎么做

多轮测试是 AI 应用里非常重要的一类测试。

### 1. 不把多轮测试当成“多个单轮拼起来”

多轮测试真正要测的是：
- 上下文延续
- 状态继承
- 工具结果引用
- 记忆命中
- 约束持续生效
- 澄清后收敛

所以多轮测试要按“完整会话”设计，而不是把每一轮独立看待。

---

### 2. 常见多轮测试模式

#### A. 上下文承接型
第一轮给背景，第二轮基于背景提问。

例如：
- 第 1 轮：记住我叫张三
- 第 2 轮：我刚刚叫什么

#### B. 澄清收敛型
先给模糊问题，再补充约束。

例如：
- 第 1 轮：帮我写个方案
- 第 2 轮：要求只改测试侧，不动主流程

#### C. 工具结果追问型
上一轮调过工具，下一轮追问结果。

例如：
- 第 1 轮：读取配置文件
- 第 2 轮：里面哪个字段控制 memory

#### D. 多轮记忆型
前几轮逐步形成可复用信息，后面验证是否被使用。

---

### 3. 多轮测试的断言方式

多轮测试实际中常断言：
- 每轮是否成功
- 第 N 轮是否引用了前面信息
- 某轮是否触发了预期工具
- 最后一轮是否满足完整任务目标
- 会话状态是否持久化

比起每轮都做重断言，更推荐：
- 前几轮做轻断言
- 关键轮做重断言
- 最后一轮做任务级断言

这样更稳。

---

## 六、一条对话重复测试在实际里怎么做

这里的“重复测试”通常有两种含义。

### 1. 稳定性重复测试
即：同一输入跑很多次，看结果是否稳定。

这在 AI 应用里非常常见，因为模型有随机性。

实际中会看：
- 通过率
- 工具调用分布
- 失败类型分布
- 输出长度分布
- 关键指标波动

例如一条 case 跑 20 次：
- 20/20 全通过：高稳定
- 17/20 通过：可用但有抖动
- 8/20 通过：高度不稳定

所以 AI 里常常不是“必须每次完全一样”，而是：

> 在重复运行中，通过率是否达到阈值。

比如：
- pass rate ≥ 95%
- 错误率 ≤ 5%

---

### 2. 幂等性/重复追问测试
即：用户重复问同一句，系统是否：
- 不自相矛盾
- 不重复乱调用工具
- 不把同一记忆无限写入
- 不越聊越漂移

这种测试很适合发现：
- session 污染
- memory 重复写入
- 工具调用失控
- prompt 漂移

---

## 七、生产里怎么设计 AI 测试，才能尽量稳定

### 1. 把断言从“文本完全一致”改成“行为与约束一致”

优先测：
- 是否命中正确流程
- 是否满足关键点
- 是否遵守约束
- 是否避免错误动作

不要过度依赖：
- 全文逐字相等

---

### 2. 把“必须同步完成”的项和“允许最终一致”的项拆开

例如：
- 同步：answer、turn、tool decision
- 最终一致：memory write、索引更新、异步回调

---

### 3. 为异步子流程单独设计测试口径

不要把它们全塞进一个 smoke case。

更建议：
- smoke：只测主流程可用性
- tools：测工具行为
- memory：测记忆检索与写入
- multi_turn：测多轮承接
- repeatability：测重复运行稳定性

---

### 4. 对 AI 测试使用“阈值思维”

尤其是重复测试和质量测试，常见做法不是单次绝对判定，而是：
- 通过率阈值
- 波动阈值
- 成本阈值
- 延迟阈值

---

### 5. 给每一类失败单独归因

例如把失败分成：
- flow failure
- tool failure
- memory failure
- answer failure
- infra failure
- timeout failure
- async eventual-consistency failure

这样报告才可用于持续优化。

---

## 八、结合当前项目，最值得补的实践点

对当前这个测试框架，后续最值得补的不是继续加更多单条断言，而是：

### 1. 增加异步等待策略
例如给 memory 类断言增加：
- 最大等待时长
- 轮询间隔
- 看到写入证据后提前结束

### 2. 区分同步断言和最终一致性断言
例如：
- `memory.expect_search_sync`
- `memory.expect_write_eventually`

或者至少在 runner 内部把 memory 写入断言改成 eventually 风格。

### 3. 增加多轮 case 格式
让一个 case 能表达：
- 第 1 轮输入
- 第 2 轮输入
- 每轮可选断言
- 最终断言

### 4. 增加重复运行模式
例如：
- 同一 case 跑 N 次
- 汇总 pass rate
- 汇总 tool/memory 波动

---

## 九、一句话总结

### 对异步功能流程
实际里通常用：
- 分层断言
- eventually 轮询
- 事件触发验证
- 最终一致性检查

### 对 AI 应用测试
实际里通常用：
- 结构测试
- 行为约束测试
- 语义覆盖测试
- 多轮测试
- 重复稳定性测试
- 阈值化评估

不要把所有东西都压成“一次请求返回后立刻完全正确”的单点断言。
