# Zhou Agent 记忆机制实现详解

## 一、总览

```
用户输入 → 检索记忆 → 注入上下文 → LLM 生成回复 → 创建 Turn
                                                      │
                                          ┌───────────┴───────────┐
                                          │ 阶段一：启发式富化      │
                                          │ (同步，在 turn 创建时)  │
                                          │ → tags                 │
                                          │ → memory_candidates    │
                                          └───────────┬───────────┘
                                                      │
                                          ┌───────────┴───────────┐
                                          │ 阶段二：LLM 富化       │
                                          │ (异步，后台线程)        │
                                          │ → reasoning_summary    │
                                          │ → 4 类记忆决策         │
                                          │   insert/update/skip   │
                                          └───────────┬───────────┘
                                                      │
                                          apply_enriched_result()
                                          → replace_turn()
                                          → 写入 session/folder 记忆
```

---

## 二、数据模型：三维度九分类

代码定义在 `src/zhou/memory/manager.py` 顶部。

### 三个维度

| 维度 | 枚举 | 设问 |
|------|------|------|
| **Scope（作用域）** | `SESSION` / `FOLDER` / `GLOBAL` | 记忆对谁可见？ |
| **Class（类别）** | `EPISODIC` / `SEMANTIC` / `PROCEDURAL` | 记忆什么类型？ |
| **Kind（检索优先级）** | `SHORT_TERM` / `LONG_TERM` / `KNOWLEDGE` | 检索时用什么窗口？ |

### 九分类矩阵

|  | EPISODIC（情景） | SEMANTIC（语义） | PROCEDURAL（程序） |
|---|---|---|---|
| **SHORT_TERM** | session 内发生的事 | session 内学到的概念 | — |
| **LONG_TERM** | folder 内的历史操作 | — | folder 内的流程/方法论 |
| **KNOWLEDGE** | — | 跨项目的知识 | — |

### MemoryRecord 核心字段

| 字段 | 含义 | 示例 |
|------|------|------|
| `scope` | 作用域 | `"session"` |
| `memory_class` | 记忆类别 | `"episodic"` |
| `kind` | 检索优先级 | `"short_term"` |
| `content` | 记忆文本 | LLM 生成的内容 |
| `metadata.memory_key` | 同概念唯一标识 | `"session:episodic:abc123"` |
| `metadata.revision` | 版本号 | `1, 2, 3…` |
| `metadata.importance` | 重要性 0-1 | `0.85` |
| `metadata.cwd` | 所属项目目录 | `"/home/user/my-project"` |
| `metadata.source` | 来源 | `"conversation"` / `"memory_model_session_semantic"` |

## 三、`turns.jsonl` 是怎么生成的

### 1. 初始 turn 由主对话链路写入

每轮对话结束后，`main._handle_turn()` 会先：

1. 收集流式返回的 `reasoning_text`
2. 用本地函数 `summarize_reasoning()` 压缩出一个短 `reasoning_summary`
3. 调 `build_turn_record()` 构造 `TurnRecord`
4. `session.append_turn(turn)` 追加写入 `.zhou/session/<id>/turns.jsonl`

也就是说，`turns.jsonl` 里的第一版 `reasoning_summary` 来自：

- **主模型流式推理文本**
- 再经本地压缩函数裁短

它不是这里再额外调一个“总结模型”得到的。

### 2. 异步 memory model 之后可能覆盖这一条 turn

后台 worker 跑完后，会重新构造 enriched turn：

- 若 `output.reasoning_summary` 有值，就优先使用它
- 若没有，就保留原来的 `job.turn.reasoning_summary`

然后 `apply_enriched_result()` 里调用 `session.replace_turn(turn)`，把 `turns.jsonl` 中同时间戳的那一条替换掉。

所以，最终 `turns.jsonl` 里的 `reasoning_summary` 可能来自两种来源：

1. 主模型流式推理压缩版
2. memory model 返回的结构化摘要版

## 四、两阶段记忆管道

### 阶段一：同步启发式富化

**代码位置**：`src/zhou/memory/enrichment.py` → `src/zhou/session.py`

**发生时机**：每轮对话结束时创建 `TurnRecord` 的瞬间。

**入口**：`session.py` 的 `build_turn_record(auto_enrich=True)`

**三个纯函数**：

| 函数 | 输入 | 输出 | 算法 |
|------|------|------|------|
| `derive_turn_tags` | 用户输入 + AI 回复 + 推理摘要 + 工具调用 | ≤8 个标签 | 正则匹配路径片段、英文关键词、工具名 |
| `derive_memory_candidates` | 同上 | ≤6 条候选记忆 | 正则匹配包含“路径/配置/约定/必须/优先”等信号词的句子 |
| `normalize_string_list` | 标签/候选列表 | 去重后的字符串列表 | 大小写折叠去重 |

**关键特性**：
- 纯文本正则，不调用 LLM
- 速度快，作为 turn 的初步 enrichment
- 结果直接写入 `TurnRecord.tags` 和 `TurnRecord.memory_candidates`

---

### 阶段二：异步 LLM 富化

**代码位置**：`src/zhou/memory/model.py` + `src/zhou/main.py`

**发生时机**：turn 落盘后，提交 `MemoryModelJob` 到后台 daemon 线程。

#### 完整调用链

```
main._handle_turn()
  │
  ├─ 1. 检索 4 类已有记忆（作为 related_memories 传给 memory model）
  │     existing_session_episodic    ← search_memory(turn.user, SESSION, SHORT_TERM)
  │     existing_session_semantic    ← search_memory(turn.assistant, SESSION, SHORT_TERM)
  │     existing_folder_procedural   ← search_memory(reasoning_summary, FOLDER, LONG_TERM)
  │     existing_folder_episodic     ← search_memory(session_id, FOLDER, LONG_TERM)
  │
  ├─ 2. 构造 MemoryModelJob（含 turn + 4 类已有记忆）
  │
  ├─ 3. worker.submit(job, callback=apply_enriched_result)
  │      │
  │      └─ MemoryModelWorker._run() 在 daemon 线程中:
  │            │
  │            ├─ client.generate_turn_enrichment(job)
  │            │     │
  │            │     ├─ build_memory_model_prompt(job)  → JSON prompt
  │            │     ├─ POST {base_url}/chat/completions
  │            │     └─ parse_memory_model_output(response)
  │            │           → MemoryModelOutput {
  │            │               reasoning_summary,
  │            │               tags,
  │            │               memory_candidates,
  │            │               session_episodic,
  │            │               session_semantic,
  │            │               folder_procedural,
  │            │               folder_session_episodic,
  │            │             }
  │            │
  │            ├─ build_turn_record(..., auto_enrich=False)
  │            │     → 用 LLM 产出的 tags/memory_candidates/reasoning_summary
  │            │       构造 enriched turn
  │            │
  │            └─ job.callback(EnrichedTurnResult(turn, output))
  │
  └─ 4. 如果 memory model 未启用:
        memory.write_session_turn(cwd, turn)
```

#### MemoryDecisionDraft 结构

```python
@dataclass
class MemoryDecisionDraft:
    decision: str = "skip"          # insert | update | skip
    target_memory_key: str = ""     # update 时指向旧记忆
    content: str = ""               # 要写入的记忆文本
    importance: float = 0.0
    reason: str = ""
```

#### Memory model 的职责

它不是主对话模型，而是独立的一套后台记忆模型，职责是：

1. 补或改 `reasoning_summary`
2. 修正 tags / memory_candidates
3. 为四类记忆分别做 `insert / update / skip` 决策

## 五、mem0 / mem 框架在这里做了什么

### 1. mem0 不是当前主要“记忆决策者”

当前 zhou 项目里，真正负责“该不该写什么记忆”的，是自己的 `memory model`。

mem0 主要承担：

- 向量存储接口
- 检索接口
- 嵌入与 Qdrant 对接

### 2. mem0 的 LLM 配置是什么

在 `Mem0MemoryManager.__init__()` 里，mem0 的内部 LLM 被配置成：

- provider: `deepseek`
- model: `settings.deepseek_model`

embedder 则是：

- provider: `huggingface`
- model: `settings.embedding_model`

所以要区分：

- **LLM 抽取模型**：DeepSeek
- **Embedding 模型**：BGE / HuggingFace 本地模型

### 3. 为什么之前会看到英文抽取

以前这里调用 `self.client.add(... infer=True)` 时，mem0 会再做一次自己的抽取/改写。

而 mem0 的抽取语言风格不受你自定义的 `MEMORY_MODEL_SYSTEM_PROMPT` 严格约束，所以容易出现：

- 内容被重新概括
- 偏英文
- 与你期望的中文结构化输出不一致

### 4. 当前代码已关闭 mem0 的二次抽取

现在代码是：

- `infer=False`

这意味着：

## mem0 现在不再自己抽取
而是：

## zhou 自己的 memory model 决定写入内容

## 六、谁在写 session / folder / global 记忆

这一节是理解“为什么现在 session 多、folder 少、global 更少”的关键。

### A. Session 记忆

#### 写入者 1：fallback 路径 `write_session_turn()`
这是最稳定、最保底的一条路径。

它固定写入：

- `scope = SESSION`
- `kind = SHORT_TERM`
- `memory_class = EPISODIC`

也就是说，只要走这条路径，产物一定是：

## `session + short_term + episodic`

#### 写入者 2：memory model 的 `session_episodic`
如果 memory model 对本轮给出：

- `session_episodic.decision in {insert, update}`

就会按 versioned memory 写入 session episodic。

#### 写入者 3：memory model 的 `session_semantic`
如果 memory model 认为本轮产生了新的会话级知识，会写：

- `session + short_term + semantic`

---

### B. Folder 记忆

Folder 没有独立的简单保底写入函数。

它只能依赖 memory model 给出的两个分支：

1. `folder_procedural`
2. `folder_session_episodic`

并且当前代码里，folder 写入的前置条件是：

## `session_memory_changed == True`

注意这是当前真实代码，不再是旧文档里的 `session_episodic_changed`。

也就是说：

- 只要本轮 session 层任一有效记忆写入发生
  - `session_episodic`
  - 或 `session_semantic`
- 就允许继续写 folder

但是如果本轮 session 层没有任何有效 insert/update，那么 folder 仍不会写。

所以 folder 本质上是：

## 依赖 session 先有有效 enrichment，再做项目层提升

---

### C. Global 记忆

当前主对话自动链路里，global 不参与每轮自动写入。

`search_all_scopes()` 会查 global：

- `scope = GLOBAL`
- `kind = KNOWLEDGE`

但写入 global 的主要接口是：

- `write_global_knowledge()`

它更像显式知识沉淀接口，而不是“每轮自动把 turn 提升到 global”。

所以你会看到：

- session 很活跃
- folder 偏少
- global 最少

这是当前设计使然。

---

## 七、为什么 session memory 里几乎全是 `episodic + short_term`

这是当前系统最容易观察到的现象。

### 原因 1：fallback 路径写死了这个分类

`write_session_turn()` 本身就是：

- `SESSION`
- `SHORT_TERM`
- `EPISODIC`

它是保底路径，因此数量天然最多。

### 原因 2：`session_semantic` 没有保底

语义记忆只能由 memory model 明确判定后写入。

很多轮次只是：

- 问一个问题
- 跑几个工具
- 得到一个结果

这更像“发生了什么”，而不是“学到了什么稳定知识”。

因此 memory model 往往更容易产出：

- episodic

而不是：

- semantic

### 原因 3：folder / global 都没有同等级别的高频兜底

结果就是数量分布天然偏向：

1. session
2. short_term
3. episodic

这不是偶然，是代码结构决定的结果。

## 八、`apply_enriched_result()` 的真实当前写入逻辑

当后台线程完成 LLM 调用后，通过 callback 回到主线程执行。

```
apply_enriched_result(session, memory, result)
  │
  ├─ 1. session.replace_turn(enriched_turn)
  │     → 用富化后的 turn 替换 turns.jsonl 中对应条目
  │
  ├─ 2. 处理 session_episodic
  │     ├─ 若 decision ∈ {insert, update} 且有内容
  │     │     → versioned 写入 SESSION + SHORT_TERM + EPISODIC
  │     │     → session_memory_changed = True
  │     └─ 否则
  │           → fallback: write_session_turn()
  │
  ├─ 3. 处理 session_semantic
  │     ├─ 若 decision ∈ {insert, update} 且有内容
  │     │     → versioned 写入 SESSION + SHORT_TERM + SEMANTIC
  │     │     → session_memory_changed = True
  │
  ├─ 4. 处理 folder_procedural
  │     ├─ 前提：session_memory_changed == True
  │     └─ 满足且 decision 有效时
  │           → versioned 写入 FOLDER + LONG_TERM + PROCEDURAL
  │
  └─ 5. 处理 folder_session_episodic
        ├─ 前提：session_memory_changed == True
        └─ 满足且 decision 有效时
              → versioned 写入 FOLDER + LONG_TERM + EPISODIC
```

### 版本化写入机制

`memory_key` 格式：

- `scope:memory_class:decision_key`

例如：

- `session:episodic:xxx`
- `folder:procedural:yyy`

每次写同 key 时 revision 自增。

检索时 `collapse_memory_result()` 会：

1. 按 `memory_key` 分组
2. 每组只保留最新 revision
3. 无 `memory_key` 的记录直接保留

这就是“同一概念保留最新版”的机制。

## 九、记忆检索

### 每轮对话开始时的检索

在 `_handle_turn()` 中调用：

```python
search_results = memory.search_all_scopes(user_input, cwd, session_id)
```

### 三层检索策略

```python
def search_all_scopes(self, query, cwd, session_id):
    return {
        "session": search_memory(query, scope=SESSION, kind=SHORT_TERM),
        "folder":  search_memory(query, scope=FOLDER,  kind=LONG_TERM),
        "global":  search_memory(query, scope=GLOBAL,  kind=KNOWLEDGE),
    }
```

| 层级 | 检索内容 | 过滤条件 | Kind |
|------|---------|---------|------|
| Session | 当前会话相关记忆 | `run_id = session_id` | SHORT_TERM |
| Folder | 当前项目目录相关记忆 | `cwd = 当前目录` | LONG_TERM |
| Global | 跨项目知识 | 无 cwd 限制 | KNOWLEDGE |

### 为什么 `/memory session` 体感上也几乎只看到 short-term

因为：

1. session 主产物本来就主要写 `SHORT_TERM`
2. `/memory session` 展示的是 session scope 的检索结果
3. 当前 session scope 下真正大量存在的就是 short-term episodic

所以你会自然感觉：

## session memory 几乎全是 episodic / short-term

## 十、为什么 folder 往往更少

这是一个结构性结果：

### session 有保底写入
- `write_session_turn()`

### folder 没保底写入
- 只能靠 memory model 产出 `folder_*` 决策
- 且要先满足 `session_memory_changed == True`

所以只要 memory model：

- 没判断出稳定的项目级经验
- 或这一轮 session 没有有效记忆更新

folder 就不会增长。

这也是为什么你之前会看到：

- `/memory session` 很快累积到多条
- `/memory folder` 很长时间还是 0

## 十一、全链路真实责任分工

### 主对话模型负责
- 生成回答
- 产生流式 reasoning 文本
- 形成初版 `reasoning_summary`

### `session.py` 负责
- 创建 `TurnRecord`
- 同步启发式 enrichment
- 写入 `turns.jsonl`

### `memory model` 负责
- 修正 reasoning_summary
- 判断四类记忆该不该 insert/update/skip
- 产出结构化中文 JSON

### `manager.py` 负责
- 把 memory model 的结果落库
- fallback 写 session episodic
- 检索与去重
- Qdrant / mem0 对接

### mem0 负责
- 存取接口
- embedding + vector store 对接
- 当前已不再负责二次内容抽取

## 十三、单轮对话后，哪些本地文件会变化

假设你在一个正常会话里发起一轮对话，典型会变化的本地文件有三类。

### 1. 当前 session 的 `turns.jsonl`

路径类似：

- `.zhou/session/<session_id>/turns.jsonl`

变化分两次发生：

#### 第一次：同步追加
由 `session.append_turn(turn)` 追加一条初版 turn：

- user
- assistant
- reasoning_summary（主模型流式压缩版）
- tool_calls
- tags（启发式）
- memory_candidates（启发式）

#### 第二次：异步替换
若 memory model 成功返回，会由 `session.replace_turn(enriched_turn)` 替换同一条记录中的：

- reasoning_summary
- tags
- memory_candidates

所以你看到的 `turns.jsonl` 最终内容，往往不是第一次写入时的原样，而是“被 memory model 修订过”的版本。

### 2. 当前 session 的 `meta.json`

路径类似：

- `.zhou/session/<session_id>/meta.json`

每轮至少会更新：

- `last_active_at`
- 可能还包含当前 active skill names

它不直接存记忆内容，但会被 `SessionState.save()` 更新。

### 3. 全局 raw archive 的当日日志文件

如果全局归档开着，`archive_writer.append_turn()` 会把这轮 turn 追加到一个按天切分的 JSONL 文件里：

- `<archive_root>/<YYYY-MM-DD>.jsonl`

其中内容是：

- timestamp
- session_id
- cwd
- project_name
- scope=`global_episodic_archive`
- user
- assistant
- tool_calls（可选）
- tags（可选）

注意：

## 这个 global archive 不是 Qdrant 里的 global memory
它是另一条 raw archive 链路，更像长期原始归档。

## 十四、单轮对话后，Qdrant 里会新增哪些记录

这一部分不在本地文件里，而是在 mem0 → Qdrant 里。

### 情况 A：memory model 没开，或本轮 `session_episodic` 没命中

会走 fallback：

- `write_session_turn()`

于是 Qdrant 至少新增 1 条：

- `scope = session`
- `kind = short_term`
- `memory_class = episodic`
- `source = conversation`
- `content = format_turn_memory_content(turn)`

这是最常见的“保底 session memory”。

### 情况 B：memory model 命中 `session_episodic`

新增或更新 1 条 versioned memory：

- `scope = session`
- `kind = short_term`
- `memory_class = episodic`
- `source = memory_model_session_episodic`
- metadata 含 `memory_key` + `revision`

### 情况 C：memory model 命中 `session_semantic`

再新增 1 条 versioned memory：

- `scope = session`
- `kind = short_term`
- `memory_class = semantic`
- `source = memory_model_session_semantic`

### 情况 D：memory model 命中 `folder_procedural`

前提是本轮 `session_memory_changed == True`，然后新增：

- `scope = folder`
- `kind = long_term`
- `memory_class = procedural`
- `source = memory_model_folder_procedural`

### 情况 E：memory model 命中 `folder_session_episodic`

同样前提下新增：

- `scope = folder`
- `kind = long_term`
- `memory_class = episodic`
- `source = memory_model_folder_session_episodic`

### 情况 F：global memory

当前自动主链路里默认不会因为一轮普通对话，自动新增 Qdrant 里的 global memory。

要进 global Qdrant memory，通常要走：

- `write_global_knowledge()`

所以普通轮次后：

- 本地 global archive 可能会增
- 但 Qdrant 的 global memory 通常不会自动增

## 十五、一轮对话后，`/memory` 命令会看到什么

### 1. `/memory turns`

这个命令读的是本地 `turns.jsonl`。

你会看到：

- user
- assistant
- reasoning_summary
- tool_calls
- tags
- memory_candidates

所以它反映的是：

## 本地 turn 存档视角
而不是向量库视角。

### 2. `/memory session`

这个命令查的是 Qdrant 中：

- `scope = session`
- 当前 `session_id`
- 当前 `cwd`

所以你看到的是：

## 当前会话的向量记忆视角
它不保证与 `turns.jsonl` 一一对应，因为：

- 一轮 turn 可能只写 1 条 episodic
- 也可能再写 1 条 semantic
- 老 revision 会被检索折叠
- 同一轮 turn 与最终 memory 数量并不是 1:1

### 3. `/memory folder`

这个命令查的是 Qdrant 中：

- `scope = folder`
- `cwd = 当前项目`

所以它看到的是：

## 当前项目级长期记忆视角
如果 memory model 很少做 folder 提升，这里就会长期偏少。

### 4. `/memory global`

这个命令查的是 Qdrant 中：

- `scope = global`

它不会去看 global archive 文件。

所以要注意：

## `/memory global` 与全局 JSONL archive 不是一回事

---

## 十六、文件 / Qdrant / `/memory` 三者怎么对照

这是最容易混淆的地方。

| 你看到的东西 | 数据来源 | 存放位置 | 它代表什么 |
|------|------|------|------|
| `turns.jsonl` | `session.append_turn` / `session.replace_turn` | 本地文件 | 原始会话 turn 存档 |
| `meta.json` | `SessionState.save()` | 本地文件 | 当前 session 元数据 |
| 全局 archive JSONL | `archive_writer.append_turn()` | 本地文件 | 跨项目原始归档 |
| `/memory session` | `search_memory(scope=session)` | Qdrant | 当前 session 的向量记忆 |
| `/memory folder` | `search_memory(scope=folder)` | Qdrant | 当前项目的长期记忆 |
| `/memory global` | `search_memory(scope=global)` | Qdrant | 跨项目知识记忆 |

### 一个关键事实

## `turns.jsonl` 不等于 session memory

因为：

- `turns.jsonl` 是完整对话轮次存档
- session memory 是从 turn 中提炼/筛选后进入向量库的产物

同理：

## global archive 也不等于 global memory

因为：

- global archive 是 raw JSONL 归档
- global memory 是结构化知识记忆

这两套东西目前是并行存在的。

---

## 十七、一个完整例子

假设你问：

> “把 memory manager 的 folder 写入条件修一下，并说明原因。”

一轮结束后可能发生的是：

### 本地文件
1. `turns.jsonl` 新增一条 turn
2. 随后该条 turn 被 memory model 替换成 enriched 版本
3. `meta.json` 更新时间
4. global archive 当日日志追加一条 raw turn

### Qdrant
1. fallback 或 model 写入 1 条 session episodic
2. 如果 model 判断本轮沉淀了新知识，再写 1 条 session semantic
3. 如果 model 觉得这形成了项目级经验，再写 1 条 folder procedural

### `/memory` 命令
- `/memory turns`：能看到这轮完整对话
- `/memory session`：通常至少看到 1 条与这轮相关的 episodic memory
- `/memory folder`：只有 model 真判断形成项目经验时才可能看到

---

## 十八、为什么你会感觉“本地很多，Qdrant 不一定多”

因为本地文件是“全量记账”，而向量记忆是“提炼后入库”。

### 本地文件特点
- 每轮都会落
- 内容完整
- 更像审计日志

### Qdrant 记忆特点
- 带筛选、带分类、带 revision
- 不保证每轮多条
- 更像可检索认知层

所以一个常见现象是：

- `turns.jsonl` 已经很多轮了
- 但 `/memory folder` 仍然很少

这完全可能，而且符合当前设计。

## 二十、最短结论速查

| 问题 | 答案 |
|------|------|
| `turns.jsonl` 里的 `reasoning_summary` 谁做的？ | 先由主对话模型流式 reasoning 压缩生成，后续可能被 memory model 覆盖 |
| mem0 英文抽取用什么模型？ | 之前用的是配置给 mem0 的 DeepSeek LLM，不是 embedding 模型；现在已关闭 mem0 自己抽取 |
| 为什么 session memory 全是 `episodic + short_term`？ | 因为 fallback 路径 `write_session_turn()` 固定就写这个分类，且它是系统里最强的保底写入路径 |
| folder 为什么少？ | 没有保底路径，只能依赖 memory model 决策，并且要先有 session 层有效更新 |
| global 为什么更少？ | 当前不是每轮自动写入，主要是显式知识沉淀接口 |
| 本地 `turns.jsonl` 和 `/memory session` 一样吗？ | 不一样。前者是原始 turn 存档，后者是 session scope 的向量记忆 |
| global archive 和 `/memory global` 一样吗？ | 不一样。前者是 raw JSONL 归档，后者是 global scope 的结构化知识记忆 |
