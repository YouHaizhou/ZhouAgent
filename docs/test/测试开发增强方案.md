# ZhouAgent 测试开发特色增强方案

## 1. 文档目标

本文档聚焦一个问题：

> 如何在当前 ZhouAgent 项目上补出更明显的“测试开发”特色，让项目在测试开发岗位面试里更有辨识度。

这里的目标不是把项目改造成纯测试框架，而是围绕 Agent 系统的特点，补出：

- 可测试性
- 可回归性
- 可观测性
- 可复现性
- 自动化质量保障能力

---

## 2. 为什么这个项目适合补测试开发特色

当前 ZhouAgent 已经具备很多适合测试开发延展的基础条件：

### 2.1 分层清晰，天然适合测试分层

当前代码已经拆分为：

- `main`：总编排
- `core`：配置、命令、LLM 通信、异常
- `session`：状态与持久化
- `skills`：技能发现与 prompt 注入
- `tools`：MCP 工具发现与调用
- `memory`：检索、回写、异步富化
- `tui`：展示层

这意味着可以自然构建：

- 单元测试
- 组件测试
- 集成测试
- 回归测试

### 2.2 外部依赖丰富，适合体现 mock / stub 能力

项目依赖：

- 主对话模型 API
- memory model API
- MCP tools
- Docker / Qdrant
- 本地文件系统

这非常适合展示测试开发的经典能力：

- 依赖隔离
- mock / stub
- 可控测试环境
- 不稳定外部系统替身设计

### 2.3 已有结构化运行数据，适合往 trace / replay 演进

当前项目已经有：

- session turns
- memory model log
- mcp server log
- 全局 archive

这些数据很适合进一步发展为：

- trace 日志
- record / replay
- 行为回归验证
- benchmark 评测

---

## 3. 最值得补的 5 个测试开发方向

## 3.1 Prompt / Tool / Memory 回归测试框架

### 目标

设计一套结构化 case，验证 Agent 在给定输入下的关键行为，而不是只比较最终字符串是否完全一致。

### 为什么这很重要

LLM 系统天然存在非确定性，如果只做全文精确比较，测试容易脆弱且噪声高。更合理的断言方式应该是“结构化断言”。

### 建议断言维度

- 是否触发了指定 tool
- 是否生成了 `reasoning_summary`
- 回答中是否包含关键结论
- 回答中是否不包含错误关键词
- 是否命中了某类 memory
- 是否写入了特定 memory scope / class
- 是否符合输出结构约束

### 面试价值

这能体现：

- 你理解 LLM 系统测试与传统接口测试的不同
- 你知道如何为非确定性系统设计稳定断言
- 你能建设可回归的行为验证体系

---

## 3.2 全链路 Trace / Event Log

### 目标

为每轮对话沉淀结构化 trace，支持问题定位、链路分析和后续评测。

### 建议记录字段

- `trace_id`
- `session_id`
- `user_input`
- `selected_skills`
- `tools_available`
- `tool_calls`
- `tool_call_count`
- `memory_hits`
- `memory_write_actions`
- `latency_ms`
- `status`
- `error`

### 面试价值

这体现：

- 可观测性建设能力
- 故障定位能力
- 数据驱动优化能力
- 测试与生产问题联动思维

---

## 3.3 Mock / Stub 测试模式

### 目标

让项目在不依赖真实外部系统的情况下也能稳定运行测试。

### 优先 mock 的点

- `LlmClient.respond_turn()`
- `MemoryModelClient.generate_turn_enrichment()`
- `tools.call_tool()`
- `memory.search_memory()` / `search_all_scopes()`

### 价值

这可以体现：

- 测试分层意识
- 外部依赖隔离能力
- 让测试具备可重复性和稳定性

---

## 3.4 Record / Replay 机制

### 目标

记录真实一次执行链路，并在后续可控地重放，用于：

- 缺陷复现
- 回归验证
- 版本对比
- 问题定位

### 可记录内容

- 输入
- tools 调用
- tool 返回
- 主模型输出
- memory model 输出
- turn 最终结果

### 面试价值

这是非常典型的平台型测试开发能力，特别适合讲“线上问题如何复现、版本修改如何回归验证”。

---

## 3.5 Benchmark / Evaluation 批量评测

### 目标

让 Agent 能批跑一组 case，并输出质量报告。

### 建议指标

- case 总数
- 通过数 / 失败数
- 平均耗时
- tool 成功率
- memory 写入成功率
- 回答格式合规率
- 回归版本对比

### 面试价值

体现你不是只会写测试，而是在做“质量平台化能力”。

---

## 4. 如果时间有限，最推荐落地的 3 个功能

## 4.1 结构化回归测试

这是优先级最高的能力。

### 建议落地内容

- 新增 case 目录
- 设计 case 文件格式
- 支持批量 runner
- 支持行为断言

### 示例断言

- `tool_called`
- `answer_contains`
- `answer_not_contains`
- `reasoning_summary_exists`
- `memory_scope_written`

---

## 4.2 Trace 日志

### 建议落地内容

每轮对话生成结构化 trace，例如：

```json
{
  "trace_id": "...",
  "session_id": "...",
  "user_input": "...",
  "tool_calls": ["filesystem.read_file"],
  "memory_hits": {
    "session": 2,
    "folder": 1,
    "global": 0
  },
  "status": "success"
}
```

### 收益

- 回归跑批时更容易定位失败点
- 面试时更容易展示“可观测性”能力

---

## 4.3 Mock + Replay

### 建议落地内容

增加测试模式：

- mock 主模型输出
- mock memory model 输出
- mock tool 结果
- 支持 replay 某条历史 case

### 收益

- 外部依赖不稳定时测试仍可跑
- 能把一次真实问题固定成回归用例

## 6. Case 设计建议

## 6.1 Case 不建议做全文精确匹配

因为 Agent / LLM 输出存在波动，所以更适合：

- 关键词断言
- 工具行为断言
- 结构化字段断言
- 副作用断言

## 6.2 推荐 case 结构

```json
{
  "name": "read_agent_file_case",
  "input": "读取 Agent.md 的内容",
  "mocks": {
    "tools": {
      "filesystem.read_file": "# Agent Profile ..."
    }
  },
  "assertions": {
    "tool_called": ["filesystem.read_file"],
    "answer_contains": ["Agent.md", "Agent Profile"],
    "reasoning_summary_exists": true
  }
}
```

## 6.3 建议覆盖的 case 分类

### 命令类
- `/skills`
- `/tools`
- `/memory help`

### tools 类
- 读取文件
- 调用 shell
- 参数异常
- tool timeout

### memory 类
- search_all_scopes 命中
- memory model 回写
- procedural / semantic 分类验证

### dialog 类
- 无工具直接回答
- 需要工具后回答
- 回答缺失 fallback

## 8. Trace 设计建议

建议每轮生成：

```json
{
  "trace_id": "uuid",
  "session_id": "...",
  "phase": [
    "memory_search",
    "prompt_build",
    "llm_respond",
    "tool_call",
    "turn_persist",
    "memory_async_submit"
  ],
  "tool_calls": [...],
  "memory_results": {...},
  "final_status": "success"
}
```

这样后续可以支持：

- trace 检索
- 回归失败对比
- 批量统计

## 9. 最适合面试时讲的亮点表达

### 9.1 质量保障体系角度

> 我没有把测试停留在单元测试，而是围绕 Agent 的 Prompt、Tool、Memory 三条主链路，设计了结构化回归测试、mock 测试模式、trace 日志和 replay 机制，让这个系统既能测、又能回归、还能复现问题。

### 9.2 非确定性系统测试角度

> 对 LLM 系统不能简单做全文字符串相等断言，所以我重点设计的是行为断言和结构化断言，例如工具调用、关键字段、memory 副作用、输出格式合规等。

### 9.3 平台化能力角度

> 我希望把 Agent 的质量能力平台化，所以会补 case runner、批量 benchmark、trace 和 replay，这样修改代码之后可以自动判断行为是否退化。

---

## 10. 三天内可落地的最小方案

如果面试时间很近，建议不要铺太大，优先做一版“小而完整”的方案。

### Day 1
- 设计 `agent_cases` case 格式
- 做最小 runner
- 支持 `answer_contains` / `tool_called`

### Day 2
- 增加 mock 主模型 / mock tool
- 跑 5~10 个典型 case
- 输出 pass/fail 报告

### Day 3
- 增加 trace 日志
- 准备面试材料：为什么这样设计、如何做断言、如何 mock、如何回归

---

## 11. 结论

对当前 ZhouAgent 来说，最能体现测试开发特色的不是“补几个 pytest”，而是建设一套：

- 结构化回归测试
- Mock / Stub
- Trace
- Replay
- Benchmark

其中优先级最高的是：

1. `Agent 行为回归测试框架`
2. `Mock 测试模式`
3. `全链路 Trace`

这是最容易被测试开发岗位面试官认可的一组增强方向。
