# 单轮对话回滚方案（简要版）

## 1. 目标

本方案只解决一个问题：

> 当用户对“上一轮对话”产生的修改不满意时，可以在终端中执行一次回滚，尽量恢复到这一轮执行前的状态。

当前阶段只支持：

- **回滚上一轮**
- **单项目内回滚**
- **以本地文件、会话、记忆为主要范围**

不追求一开始就支持：

- 多轮任意跳转回滚
- 外部系统副作用的完全恢复
- 所有 MCP 工具的精确语义回滚

---

## 2. 当前已具备的基础

`.zhou` 下目前已经有两类基础信息：

### 2.1 session

用于记录：

- 对话内容
- 工具调用信息
- 当前会话状态

### 2.2 logs

用于记录：

- 记忆处理日志
- 工具调用相关日志
- 部分工具副作用线索

这些内容已经可以作为“回滚能力”的前置基础，但还不够直接支持恢复。

因此建议在 `.zhou` 下新增一类专门面向回滚的目录：

- `backup/`

---

## 3. 建议的目录结构

建议在项目 `.zhou` 下增加如下结构：

```text
.zhou/
  session/
  logs/
  backup/
    last-turn/
      meta.json
      files/
      memory/
```

含义如下。

### 3.1 `backup/last-turn/meta.json`

记录上一轮回滚所需的元信息，例如：

- 本轮时间
- session id
- turn id
- 用户输入
- 助手输出摘要
- 本轮是否调用工具
- 本轮涉及哪些文件
- 本轮涉及哪些记忆更新
- 本轮回滚状态

这个文件相当于“上一轮回滚索引”。

### 3.2 `backup/last-turn/files/`

用于保存**本轮执行前**的文件备份。

适用于：

- 文件修改前备份原内容
- 文件删除前备份原内容
- 新增文件则只记录路径，不一定需要备份内容

### 3.3 `backup/last-turn/memory/`

用于保存**本轮记忆更新后，为了后续回滚所需的记忆信息**。

例如可记录：

- 本轮新增了哪些 memory revision
- 本轮更新了哪些 memory key
- 回滚时需要失效或恢复哪些记忆状态

---

## 4. 备份更新时机

你现在的想法是对的，建议拆成两个时间点。

## 4.1 工具执行前

如果本轮工具调用会修改文件，那么在真正执行前，需要先写入回滚备份。

分两种情况：

### 情况 A：涉及文件修改

例如：

- 改文件
- 删文件
- 重命名文件
- 新建文件

这时需要：

- 记录受影响路径
- 对原文件内容做备份
- 记录操作类型（新增 / 修改 / 删除 / 重命名）

### 情况 B：不涉及文件修改

例如：

- 只读文件
- 搜索
- 列目录
- 纯查询类工具

这时不需要做文件备份，但仍然可以在 `meta.json` 中记录：

- 本轮调用了哪些工具
- 工具是否产生了副作用

---

## 4.2 当轮记忆更新后

除了文件，还要处理记忆回滚。

因为你当前有记忆版本机制，所以建议在本轮记忆写入完成后，把回滚需要的信息写入：

- `backup/last-turn/memory/`
- 以及 `meta.json`

至少要记录：

- 本轮写入了哪些记忆
- 对应的 memory key
- 对应 revision
- 属于哪类记忆（session / folder / global）
- 回滚时如何处理

当前阶段不一定要直接删除旧记忆，更适合：

- 标记本轮 revision 为已回滚
- 或恢复到上一 revision

---

## 5. 回滚时的终端流程

建议后续提供一个终端命令，例如：

```text
/rollback
```

当前只处理“上一轮回滚”，所以流程可以尽量简单。

## 5.1 执行流程

1. 读取 `.zhou/backup/last-turn/meta.json`
2. 检查是否存在可回滚的上一轮记录
3. 提示用户本轮将回滚哪些内容
   - 哪些文件会恢复
   - 哪些新增文件会删除
   - 哪些记忆会撤回
4. 用户确认后执行回滚
5. 回滚完成后更新回滚状态
6. 输出结果摘要

## 5.2 回滚动作

建议最小范围包括：

### 文件侧

- 修改文件：恢复备份内容
- 删除文件：从备份恢复
- 新增文件：删除新增文件

### 会话侧

- 将本轮 turn 从当前会话记录中撤掉
- 如果本轮有 enrich 后替换，也要恢复到前一状态

### 记忆侧

- 将本轮写入的 memory revision 标记为无效
- 或恢复到上一 revision

---

## 6. 终端输出建议

执行 `/rollback` 后，输出不需要复杂，建议分三类。

## 6.1 回滚前提示

例如：

- 检测到可回滚上一轮
- 将恢复 2 个文件
- 将删除 1 个新增文件
- 将撤回 3 条记忆更新
- 是否继续

## 6.2 回滚成功输出

例如：

- 上一轮回滚完成
- 已恢复文件：...
- 已删除新增文件：...
- 已撤回记忆：...
- 当前会话已退回到上一轮状态

## 6.3 回滚失败或部分失败输出

例如：

- 回滚部分完成
- 文件恢复成功，记忆撤回失败
- 某路径恢复失败：...
- 建议用户手动检查

也就是说，输出应该重点回答两件事：

- **做了什么**
- **有没有剩余风险**

---

## 7. 代码如何融入当前项目

建议不要单点乱加，而是按职责融入。

## 7.1 新增一个 rollback/backups 相关模块

建议后续新增一组专门代码，例如：

- `src/zhou/rollback.py`
- 或 `src/zhou/rollback/`

职责包括：

- 生成上一轮备份目录
- 写入 `meta.json`
- 保存文件备份
- 保存记忆回滚信息
- 执行回滚流程

## 7.2 融入工具执行链路

在“真正执行会修改文件的工具前”接入备份逻辑。

也就是：

- 先判断本次 tool call 是否可能改文件
- 如果会改，就先备份，再执行工具

这部分应接在工具执行主链路附近，而不是零散放在别处。

## 7.3 融入记忆写入链路

在本轮记忆更新完成后，把相关 revision 信息写入回滚记录。

## 7.4 融入命令入口

在终端命令体系里新增：

- `/rollback`

作为用户触发入口。

---

## 8. 当前阶段最合适的实现策略

不建议一步做到“精确回滚所有副作用”。

当前更适合的方案是：

1. **只支持上一轮**
2. **先覆盖文件、会话、记忆三类核心状态**
3. **以受影响文件备份为主，不做全项目备份**
4. **配合结构化 `meta.json` 记录回滚信息**

这是一个相对简单、可落地、也方便后续演进的方案。

---

## 9. 简要结论

基于你当前项目结构，单轮对话回滚功能建议这样设计：

- 保留现有 `.zhou/session` 和 `.zhou/logs`
- 新增 `.zhou/backup/last-turn/`
- 在工具执行前做文件侧备份
- 在记忆更新后记录记忆侧回滚信息
- 通过 `/rollback` 只回滚上一轮
- 输出结果聚焦“恢复了什么、还有什么风险”

这个方案适合作为 Zhou 当前阶段的第一版回滚能力。
